# SRT Template

**Шаблон хранилища знаний по методу SRT** (Systems-Roles-Table) — организация проекта через таблицу 3×3 систем и ролей с интеграцией First Principles Framework (FPF).

## Для кого этот шаблон

- **Проектные команды** — организация знаний о продукте
- **Стартапы** — структурирование описания проекта
- **Разработчики** — документация архитектуры и процессов
- **Исследователи** — управление базой знаний

## Что такое SRT-метод

SRT (Systems-Roles-Table) — это способ организации знаний через **таблицу 3×3**, которая определяет **9 семейств документов** + **1 метасемейство** (управление хранилищем).

### Таблица 3×3: 9 семейств документов

```
                     Meaning          Architecture      Operations
                     (Зачем?)         (Как устроено?)   (Как работает?)
┌────────────────────────────────────────────────────────────────────┐
│ Supersystem        F1               F2                F3           │
│ (Надсистема)       Контекст         Окружение         Взаимодействие
│                    и рынок          и интерфейсы      и партнёрства│
├────────────────────────────────────────────────────────────────────┤
│ TargetSystem       F4               F5                F6           │
│ (Целевая система)  Требования       Архитектура       Реализация   │
│                    и ценность       продукта          и процессы   │
├────────────────────────────────────────────────────────────────────┤
│ CreationSystem     F7               F8                F9           │
│ (Система создания) Принципы         Платформа         Команда      │
│                    и экономика      и инструменты     и методология│
└────────────────────────────────────────────────────────────────────┘
```

### F0: Метасемейство — Управление хранилищем

```
┌────────────────────────────────────────────────────────────────────┐
│ Management (F0)    Логика           Структура         Процессы     │
│ (Управление)       хранилища        хранилища         работы       │
└────────────────────────────────────────────────────────────────────┘
```

**F0 — это 10-е семейство**, которое управляет остальными девятью. Здесь находится логика организации знаний, стандарты оформления и процессы работы с хранилищем.

### Все 10 семейств документов

| Семейство | Система | Роль | Назначение | Целевая аудитория |
|-----------|---------|------|------------|-------------------|
| **F0** | Management | Meta | Управление хранилищем | AI-агенты, администраторы |
| **F1** | Supersystem | Meaning | Контекст, рынок, ценность | Внешние проектные роли, инвесторы |
| **F2** | Supersystem | Architecture | Окружение, интерфейсы | Пользователи, аналитики |
| **F3** | Supersystem | Operations | Взаимодействие, партнёрства | Партнёры, регуляторы |
| **F4** | TargetSystem | Meaning | Требования, ценность продукта | Product Owner, заказчики |
| **F5** | TargetSystem | Architecture | Архитектура продукта | Архитекторы, разработчики |
| **F6** | TargetSystem | Operations | Реализация, процессы | Разработчики, QA |
| **F7** | CreationSystem | Meaning | Принципы, экономика | Фаундеры, инвесторы |
| **F8** | CreationSystem | Architecture | Платформа, инструменты | DevOps, архитекторы |
| **F9** | CreationSystem | Operations | Команда, методология | Менеджеры, HR |

# Быстрый старт: Разворачивание хранилища для вашего проекта

## Шаг 1: Копирование шаблона

### Вариант A: Fork на GitHub

1. Нажмите **Fork** на странице репозитория
2. Укажите имя для вашего репозитория
3. Склонируйте форк локально:

```bash
git clone https://github.com/YOUR_USERNAME/YOUR_REPO_NAME.git
cd YOUR_REPO_NAME
```

### Вариант B: Локальное копирование

```bash
git clone https://github.com/TserenTserenov/srd-template.git my-project
cd my-project
rm -rf .git
git init
```

## Шаг 2: Проверка FPF

Убедитесь, что FPF загружен:

```bash
# Проверить наличие FPF-Spec.md
ls -la .fpf/

# Должны быть файлы:
# - INDEX.md (~250 строк)
# - FPF-Spec.md (~43000 строк)
# - FPF-Readme.md
```

### Обновление FPF (если нужно)

```bash
curl -sL "https://raw.githubusercontent.com/ailev/FPF/main/FPF-Spec.md" \
  -o .fpf/FPF-Spec.md
```

---

## Шаг 3: Создание описания проекта

Создайте файл `PROJECT_DESCRIPTION.md` в корне репозитория. **Используйте шаблон из файла** [templates/PROJECT_DESCRIPTION_TEMPLATE.md](templates/PROJECT_DESCRIPTION_TEMPLATE.md).

### Что нужно для начала работы

Для начала работы от вас необходимо **как минимум краткое описание своей задумки**, в том числе было бы хорошо получить указание на:
- **Целевую систему** — что создаём
- **Надсистему** — в какой системе будет работать целевая система
- **Систему создания** — кто и как создаёт

Если сразу не получается сформулировать системы чётко — сделайте описание по следующему шаблону:

### Структура описания:

1. **Общее описание** — название, краткое описание, команда
2. **Целевая система (TargetSystem)** — что создаём, для кого, какую проблему решает
3. **Надсистема (Supersystem)** — система клиента, куда входит целевая система, а также описание рынка, конкурентов, других проектных ролей и ограничений
4. **Система создания (CreationSystem)** — команда, инструменты, методология
5. **Первичные названия систем** — предложения по именованию

---

## Шаг 4: Экспертиза описания (с AI)

После заполнения `PROJECT_DESCRIPTION.md` запустите экспертизу с Claude Code:

```
Прочитай PROJECT_DESCRIPTION.md и проведи экспертизу на основе FPF и SRT:

1. Проверь чёткость определения целевой системы (паттерны A.1, A.7)
2. Проверь полноту описания надсистемы (паттерн A.1.1)
3. Проверь реалистичность системы создания
4. Оцени предложенные названия систем
5. Дай рекомендации по улучшению

Используй паттерны FPF для обоснования критики.
```

> **Важно:** На следующем шаге 5 вам потребуется **определиться с тремя системами** (целевая, надсистема, система создания), поскольку структура хранилища строится исходя из этого. При необходимости более глубокой проработки можно отдельно поработать с FPF (`.fpf/FPF-Spec.md`).

### Критерии качества описания

| Критерий | Хорошо ✅ | Плохо ❌ |
|----------|----------|---------|
| Целевая система | "Мобильное приложение для учёта личных финансов" | "Крутой продукт для людей" |
| Надсистема | "Рынок личных финансов РФ, конкурент — приложение X" | "Все люди" |
| Система создания | "Команда 3 человека, React Native, 6 месяцев" | "Мы справимся" |

---

## Шаг 5: Принятие решения о названиях систем

После экспертизы **вы сами** принимаете решение о названиях:

```markdown
## Финальные названия систем

| Система | Название | Обоснование |
|---------|----------|-------------|
| TargetSystem | FinTracker | Приложение для финансов |
| Supersystem | FinMarket | Рынок личных финансов |
| CreationSystem | DevTeam | Команда разработки |
```

> ⚠️ **Важно:** Названия должны быть осмысленными и отражать суть вашего проекта.

---

## Шаг 6: Минимальное разворачивание хранилища

После определения названий запустите команду:

```
Разверни минимальную структуру хранилища для проекта:

TargetSystem: [Ваше название]
Supersystem: [Ваше название]
CreationSystem: [Ваше название]

Создай:
1. Обновлённые README для каждого раздела с новыми названиями
2. По одному документу в каждой ячейке Meaning (F1, F4, F7)
3. Глоссарий проекта в 0.Management/0.1.Meaning/
```

### Результат минимального разворачивания

```
content/
├── 0.Management/            # F0 — Управление (без изменений)
│   └── 0.1.Meaning/
│       └── project-glossary.md  # Глоссарий ВАШЕГО проекта
├── 1.[Supersystem]/         # F1-F3 (переименовано)
│   └── 1.1.Meaning/
│       └── context.md       # F1: Описание контекста
├── 2.[TargetSystem]/        # F4-F6 (переименовано)
│   └── 2.1.Meaning/
│       └── requirements.md  # F4: Основные требования
└── 3.[CreationSystem]/      # F7-F9 (переименовано)
    └── 3.1.Meaning/
        └── principles.md    # F7: Принципы разработки
```

```
Настрой автопроверки хранилища:

2. Сгенерируй первый отчёт в 0.4.AutoReports/

```

# Структура репозитория

```
srd-template/
├── README.md                    # Этот файл
├── CLAUDE.md                    # Инструкции для AI
├── CONTRIBUTING.md              # Правила участия
├── PROJECT_DESCRIPTION.md       # Ваше описание проекта (создаёте вы)
│
├── templates/                   # Шаблоны
│   └── PROJECT_DESCRIPTION_TEMPLATE.md
│
├── .fpf/                        # First Principles Framework
│   ├── INDEX.md                 # Локальные принципы
│   ├── FPF-Spec.md              # Полная спецификация
│   └── FPF-Readme.md            # Обзор FPF
│
└── content/                     # Контент по SRT (10 семейств)
    ├── 0.Management/            # F0: Метасемейство
    │   ├── 0.1.Meaning/         # Логика и принципы
    │   ├── 0.2.Architecture/    # Структура
    │   ├── 0.3.Operations/      # Процессы
    │   ├── 0.4.AutoReports/     # Автоотчёты AI
    │   ├── 0.5.ValidationSpecs/ # ТЗ для автопроверок
    │   ├── 0.6.AccumulationGuide/ # Процесс накопления
    │   ├── 0.9.Incoming/        # Входящие идеи
    │   └── 0.99.Archive/        # Архив
    ├── 1.Supersystem/           # F1-F3: Надсистема → [Ваше название]
    ├── 2.TargetSystem/          # F4-F6: Целевая система → [Ваше название]
    └── 3.CreationSystem/        # F7-F9: Система создания → [Ваше название]
```
