# Создание документов

## Назначение документа

Пошаговая инструкция по созданию новых документов в хранилище.

## Процесс создания

### Шаг 1: Определить необходимость

**Вопросы для проверки:**

- [ ] Нужен ли документ сейчас? (не "на будущее")
- [ ] Нет ли похожего документа?
- [ ] Достаточно ли информации для наполнения?

**Если ответ "нет" на любой вопрос — не создавай документ.**

### Шаг 2: Определить место в SRT

**Выбор системы:**

| Вопрос | Система |
|--------|---------|
| Об управлении хранилищем? | 0.Management |
| О внешней среде, рынке? | 1.Supersystem |
| О создаваемом продукте? | 2.TargetSystem |
| О процессе создания? | 3.CreationSystem |

**Выбор роли:**

| Вопрос | Роль |
|--------|------|
| О целях, ценностях, требованиях? | Meaning |
| О структуре, компонентах? | Architecture |
| О процессах, действиях? | Operations |

**Результат:** Путь `content/N.System/N.M.Role/`

### Шаг 3: Выбрать имя файла

**Правила:**
- Нижний регистр
- Дефис как разделитель
- Описательное имя
- Расширение `.md`

**Примеры:**

| Тема | Имя файла |
|------|-----------|
| API документация | `api-documentation.md` |
| Требования к авторизации | `auth-requirements.md` |
| Процесс деплоя | `deployment-process.md` |

### Шаг 4: Создать файл с frontmatter

```yaml
---
type: doc                        # doc, spec, process, report
status: draft                    # Начинай с draft
created: 2026-01-07
updated: 2026-01-07
system: "SystemName"
role: "RoleName"
layer: methodology               # или architecture, operations
scope: local-edge                # или project, ecosystem
target_audience: [developers]
related: []                      # Заполнишь позже
fpf_principles: []               # Заполнишь позже
fpf_patterns: []
tags: []
---

# Название документа
```

### Шаг 5: Написать контент

**Обязательные разделы:**

1. **Назначение документа** — 1-2 предложения о цели
2. **Основное содержание** — суть документа
3. **Changelog** — история изменений
4. **Связанные документы** — ссылки

**Шаблон:**

```markdown
# Название

## Назначение документа

[Краткое описание цели документа]

## [Основной раздел]

[Контент]

## Changelog

| Дата | Изменение | Автор |
|------|-----------|-------|
| 2026-01-07 | Создание | Name |

## Связанные документы

- [Документ 1](path/to/doc1.md)
```

### Шаг 6: Добавить связи

**Найти связанные документы:**

1. Документы в той же папке
2. Документы в смежных ролях (Meaning ↔ Architecture ↔ Operations)
3. Документы в родительской/дочерней системе

**Обновить frontmatter:**

```yaml
related:
  - ../0.1.Meaning/requirements.md
  - ../0.3.Operations/implementation.md
```

**Обновить связанные документы** — добавить обратную ссылку!

### Шаг 7: Применить FPF-принципы

**Проверить и указать:**

```yaml
fpf_principles:
  - minimalism        # Если документ минимален
  - practicality      # Если практически применим
  - clarity           # Если особенно понятен
fpf_patterns:
  - A.1               # Если применяется холонический подход
  - A.7               # Если важно строгое различение
```

### Шаг 8: Финальная проверка

**Чек-лист:**

- [ ] Frontmatter заполнен полностью
- [ ] Путь соответствует system/role
- [ ] Есть раздел "Назначение документа"
- [ ] Есть Changelog
- [ ] Указаны related документы
- [ ] Обратные ссылки добавлены
- [ ] Нет дублирования с существующими документами
- [ ] Соблюдены стандарты оформления

### Шаг 9: Создать PR

```bash
git checkout -b feature/add-document-name
git add content/N.System/N.M.Role/document-name.md
git commit -m "Add: document-name in System/Role"
git push -u origin feature/add-document-name
```

## Типы документов

### doc (документ)

Концептуальное описание, обзор, руководство.

**Примеры:** принципы, глоссарий, обзоры

### spec (спецификация)

Формальное описание структуры, интерфейса, модели.

**Примеры:** API-схема, модель данных, контракт

### process (процесс)

Пошаговая инструкция, workflow.

**Примеры:** процесс деплоя, создание документа

### report (отчёт)

Результат анализа, статус, метрики.

**Примеры:** еженедельный статус, аудит качества

### template (шаблон)

Образец для создания новых документов.

**Примеры:** шаблон требования, шаблон спецификации

## Частые ошибки

| Ошибка | Как избежать |
|--------|--------------|
| Документ "на будущее" | Создавай только когда есть контент |
| Нет frontmatter | Всегда начинай с YAML-блока |
| Неправильная папка | Проверь соответствие system/role |
| Нет связей | Добавь related до создания PR |
| Односторонние ссылки | Обнови связанные документы |
