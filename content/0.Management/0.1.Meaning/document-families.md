# Модель семейств документов

## Предварительные требования

> **ОБЯЗАТЕЛЬНО:** Для эффективной работы с этим хранилищем требуется понимание основ системного мышления.
>
> **Рекомендуемый материал:** ["Введение в системное мышление"](https://docs.system-school.ru/ru/personal/systems-thinking-introduction/preface) — руководство, охватывающее:
> - Различие системности и систематичности
> - Описание и воплощение систем
> - Роли и ролевое поведение
> - Системные уровни и цепочки создания
> - Системное моделирование

---

## Что такое семейство документов

**Семейство документов** — это категория документов, объединённых по двум критериям:

1. **Система** (System) — о какой части проекта документ
2. **Роль** (Role) — какой аспект системы описывает

Семейство — это не физическая папка, а **"взгляд" на знания** проекта. Один документ принадлежит одному семейству.

---

## Принцип организации: таблица 3×3

SRT-метод организует знания через **таблицу 3×3**, которая определяет **9 основных семейств** + **1 метасемейство** (управление).

### Три системы (строки) — точное описание

| Система | Английское название | Вопрос | Системное описание |
|---------|---------------------|--------|-------------------|
| **Надсистема** | Supersystem | В каком контексте? | Система, частью которой является целевая система. Включает среду, рынок, внешние проектные роли, регуляторов. Определяет внешние ограничения и возможности |
| **Целевая система** | TargetSystem | Что создаём? | Система, которую команда создаёт для изменения надсистемы. Продукт, услуга, изменение в мире. Имеет чёткие границы и функцию в надсистеме |
| **Создающая система** | CreationSystem | Кто/как создаёт? | Система, которая создаёт целевую систему. Включает команду, методы, инструменты, инфраструктуру. Её "продукт" — целевая система |

**Важно:** Это *разные системы*, не уровни детализации одной системы. Документ всегда описывает *одну* из этих трёх систем.

### Три роли (столбцы) — точное описание

| Роль | Английское название | Вопрос | Описание по FPF (A.7) |
|------|---------------------|--------|-----------------------|
| **Смысл** | Meaning | Зачем? | Интенсионал системы: цели, назначение, ценность, причины существования. Отвечает на "почему это нужно" |
| **Архитектура** | Architecture | Как устроено? | Структурное описание: компоненты, связи, интерфейсы, ограничения. Отвечает на "из чего состоит" |
| **Операции** | Operations | Как работает? | Функциональное описание: процессы, workflow, поведение во времени. Отвечает на "что происходит" |

**Строгое различение (FPF A.7):**
- Роль ≠ Метод ≠ Работа
- Описание ≠ Объект (карта ≠ территория)
- План ≠ Исполнение

## Критерии качества описания (FPF E.10.D2)

Каждый документ должен соответствовать **I/D/S-дисциплине** (Intension–Description–Specification):

| Критерий | Вопрос | Признак качества |
|----------|--------|------------------|
| **Интенсионал (I)** | Что это *по сути*? | Определение через род и видовое отличие |
| **Дескрипция (D)** | Что *наблюдаемо*? | Конкретные, проверяемые характеристики |
| **Спецификация (S)** | Что *требуется*? | Тестируемые, верифицируемые критерии |

### Уровни формальности (FPF C.2.3 F-шкала)

| Уровень | F-код | Описание | Пример |
|---------|-------|----------|--------|
| Неформальный | F0-F2 | Свободный текст, нарратив | Блог-пост, идея |
| Полуформальный | F3-F5 | Структурированный текст, таблицы | Этот документ |
| Формальный | F6-F7 | Схемы, модели, диаграммы | UML, ER-диаграммы |
| Строго формальный | F8-F9 | Формальные спецификации | Математические доказательства |

## 10 семейств документов

### F0: Management (Метасемейство)

**Назначение:** Управление самим хранилищем знаний.

F0 — **особое семейство**, которое не входит в таблицу 3×3, но управляет остальными девятью. Это "операционная система" хранилища.

| Подраздел | Назначение | Содержимое |
|-----------|------------|------------|
| F0.Meaning | Логика хранилища | Принципы, терминология, семейства |
| F0.Architecture | Структура хранилища | Папки, роли, стандарты |
| F0.Operations | Процессы работы | Workflows, автоотчёты, валидация |

**Разработчики:** Администраторы хранилища
**Целевые читатели:** AI-агенты, редакторы
**Исключения:** НЕ для конечных пользователей продукта, НЕ для инвесторов

---

### F1: Надсистема × Смысл

**Полное название:** Контекст и смысл существования

**Назначение:** Описание внешней среды, рынка, ценностного предложения.

**Системное описание:**
Документы F1 отвечают на вопрос "Зачем надсистеме нужна целевая система?" — описывают боль/потребность, которую целевая система закрывает в контексте надсистемы.

**Разработчики:** Маркетологи, Product Marketers
**Целевые читатели:** Потенциальные участники, клиенты
**Исключения:** НЕ для инвесторов (→F7), НЕ для разработчиков (→F5)

**Типичные документы:**
- Анализ рынка и конкурентов
- Ценностное предложение (Value Proposition)
- Problem Statement
- Customer Personas

---

### F2: Надсистема × Архитектура

**Полное название:** Окружение и интерфейсы

**Назначение:** Описание структуры внешней среды, с которой взаимодействует продукт.

**Системное описание:**
Документы F2 описывают *структуру надсистемы* — какие внешние системы существуют, как они связаны, где границы нашей целевой системы.

**Разработчики:** Product Owner, Системные аналитики
**Целевые читатели:** Интеграторы, партнёры
**Исключения:** НЕ для внутренней команды разработки (→F5)

**Типичные документы:**
- Карта проектных ролей
- Context Diagram (системный контекст)
- Описание внешних API и интеграций
- Boundary Analysis

---

### F3: Надсистема × Операции

**Полное название:** Взаимодействие и партнёрства

**Назначение:** Процессы взаимодействия с внешним миром.

**Системное описание:**
Документы F3 описывают *поведение надсистемы* — как она работает, какие процессы происходят вне нашей системы, как мы с ними взаимодействуем.

**Разработчики:** BD-менеджеры, PR-специалисты
**Целевые читатели:** Партнёры, регуляторы, медиа
**Исключения:** НЕ для пользователей (→F4, F6)

**Типичные документы:**
- Партнёрские соглашения
- Регуляторные требования
- Stakeholder Communication Plan
- SLA с внешними сервисами

---

### F4: Целевая система × Смысл

**Полное название:** Требования и ценность продукта

**Назначение:** Что должен делать продукт и почему это важно.

**Системное описание:**
Документы F4 описывают *интенсионал целевой системы* — её назначение, функции, критерии успеха. Это "контракт" с пользователем.

**Разработчики:** Product Owner, Визионер
**Целевые читатели:** Пользователи, участники проекта
**Исключения:** НЕ для инвесторов (→F7)

**Типичные документы:**
- Product Requirements Document (PRD)
- User Stories и Acceptance Criteria
- Definition of Done
- Vision & Scope Document

---

### F5: Целевая система × Архитектура

**Полное название:** Архитектура продукта

**Назначение:** Как устроен продукт внутри.

**Системное описание:**
Документы F5 описывают *структуру целевой системы* — из каких компонентов она состоит, как они связаны, какие технологии используются.

**Разработчики:** Архитекторы, Методологи
**Целевые читатели:** Senior-разработчики, техлиды
**Исключения:** НЕ для пользователей напрямую (→F4, F6)

**Типичные документы:**
- Architecture Decision Records (ADR)
- Component Diagrams
- Data Model / ER-диаграммы
- API Specification

---

### F6: Целевая система × Операции

**Полное название:** Реализация и процессы продукта

**Назначение:** Как продукт работает, как его использовать и поддерживать.

**Системное описание:**
Документы F6 описывают *поведение целевой системы* — как она функционирует, какие процессы в ней происходят, как с ней взаимодействовать.

**Разработчики:** Разработчики, Деканат (для образования)
**Целевые читатели:** Пользователи, Support, DevOps
**Исключения:** НЕ для инвесторов (→F7)

**Типичные документы:**
- User Guide / Руководство пользователя
- Deployment Guide
- Runbook / Troubleshooting Guide
- Release Notes

---

### F7: Создающая система × Смысл

**Полное название:** Принципы и экономика создания

**Назначение:** Зачем и на каких принципах работает команда/организация.

**Системное описание:**
Документы F7 описывают *интенсионал создающей системы* — бизнес-модель, ценности команды, причины существования организации.

**Разработчики:** Фаундеры, Бизнес-стратеги
**Целевые читатели:** Инвесторы, стратегические партнёры
**Исключения:** НЕ для пользователей (→F4), НЕ для операционной команды (→F9)

**Типичные документы:**
- Business Model Canvas
- Investment Deck
- Team Values / Culture Code
- Financial Model

---

### F8: Создающая система × Архитектура

**Полное название:** Платформа и инструменты

**Назначение:** Какие инструменты и инфраструктура используются для создания продукта.

**Системное описание:**
Документы F8 описывают *структуру создающей системы* — tech stack, инфраструктуру, инструменты разработки.

**Разработчики:** DevOps, Platform Engineers
**Целевые читатели:** Разработчики, техлиды
**Исключения:** НЕ для пользователей, НЕ для инвесторов

**Типичные документы:**
- Tech Stack Overview
- CI/CD Pipeline Description
- Infrastructure as Code
- Development Environment Setup

---

### F9: Создающая система × Операции

**Полное название:** Команда и методология

**Назначение:** Как организована работа команды.

**Системное описание:**
Документы F9 описывают *поведение создающей системы* — процессы разработки, роли, методологию, ритм работы.

**Разработчики:** HR, Engineering Managers
**Целевые читатели:** Операционная команда, новые сотрудники
**Исключения:** НЕ для пользователей, НЕ для инвесторов

**Типичные документы:**
- Team Structure
- Development Process (Scrum/Kanban)
- Onboarding Guide
- Meeting Cadence / Rituals

## Типы ролей

### Роли-разработчики (создатели документов)

**Стратегические:**
- Визионер — определяет направление развития
- Фаундер — владелец бизнес-модели
- Стратег — долгосрочное планирование

**Продуктовые:**
- Product Owner — владелец требований
- UX-дизайнер — пользовательский опыт
- Системный аналитик — требования и интеграции

**Маркетинговые:**
- Маркетолог — позиционирование и продвижение
- Контент-маркетолог — создание контента
- PR-менеджер — внешние коммуникации

**Технические:**
- Архитектор — техническая архитектура
- Разработчик — реализация кода
- DevOps — инфраструктура и деплой
- Техлид — технические решения в команде

**Операционные:**
- HR-менеджер — люди и процессы
- Администратор — операционные вопросы
- Финансист — финансовое планирование

### Роли-читатели (целевая аудитория)

**Внешние:**
- Потенциальные клиенты — ищут решение проблемы
- Инвесторы — оценивают бизнес
- Партнёры — ищут интеграции
- Регуляторы — проверяют соответствие
- Медиа — пишут о проекте

**Пользователи:**
- Новички — начинают использовать продукт
- Активные пользователи — используют регулярно
- Power Users — используют продвинутые функции

**Команда:**
- Разработчики — пишут код
- Операторы — обеспечивают работу
- Техлиды — принимают технические решения
- Новые сотрудники — онбордятся

**Руководство:**
- Фаундеры — владельцы проекта
- Руководители служб — отвечают за направления

## Правила проверки соответствия

### Критические ошибки классификации

| Ключевые слова | Правильное семейство | Типичная ошибка |
|----------------|---------------------|-----------------|
| ROI, инвестиции, оценка компании | **F7** (не F1, F4) | Путают с маркетингом или требованиями |
| Архитектура платформы, API, БД | **F8** (не F4, F5) | Путают с архитектурой продукта |
| Путь пользователя, онбординг | **F6** (не F1, F4) | Путают с маркетингом или требованиями |
| Tech stack, CI/CD | **F8** (не F5) | Путают с архитектурой продукта |
| Бизнес-модель, финансы | **F7** (не F1) | Путают с маркетингом |

### Алгоритм определения семейства

```
1. О КАКОЙ СИСТЕМЕ документ?
   ├─ О внешней среде, рынке, проектных ролях → Надсистема (F1-F3)
   ├─ О продукте, который создаём → Целевая система (F4-F6)
   ├─ О команде, инструментах, процессах создания → Создающая система (F7-F9)
   └─ О хранилище документов → Management (F0)

2. КАКОЙ АСПЕКТ описывает?
   ├─ Зачем? Цели? Ценность? → Meaning (X.1)
   ├─ Как устроено? Структура? → Architecture (X.2)
   └─ Как работает? Процессы? → Operations (X.3)

3. СЕМЕЙСТВО = Система × Роль
```

## FPF-интеграция

| Паттерн FPF | Применение в модели семейств |
|-------------|------------------------------|
| **A.1 Holonic Foundation** | Каждое семейство — холон в иерархии знаний |
| **A.1.1 Bounded Context** | Каждое семейство — ограниченный контекст со своей терминологией |
| **A.7 Strict Distinction** | Система ≠ Роль ≠ Документ; Описание ≠ Объект |
| **E.10.D2 I/D/S Discipline** | Документы содержат Интенсионал, Дескрипцию, Спецификацию |
| **C.2.3 F-шкала** | Документы имеют уровень формальности F0-F9 |
| **B.4 Canonical Evolution** | Семейства развиваются инкрементально через коммиты |

## Связь с системным мышлением

Для понимания модели семейств необходимо владеть концепциями:

| Концепция | Применение в SRT |
|-----------|------------------|
| **Системные уровни** | Надсистема → Целевая система → Подсистемы |
| **Цепочка создания** | Создающая система производит целевую систему |
| **Роли vs Исполнители** | Роль — это функция, человек может играть несколько ролей |
| **Описание vs Воплощение** | Документ описывает систему, но не является ею |

**Рекомендуемое обучение:**
[Введение в системное мышление](https://docs.system-school.ru/ru/personal/systems-thinking-introduction/preface)
