# Раздел F19. Repository-management

## Назначение раздела

Раздел **Repository-management** — это **19-е семейство документов (F19)**, которое выполняет две функции:
1. **Метасистема хранилища** — управляет логикой организации знаний, процессами и автоматизацией
2. **Мосты между ядрами** — описывает связи между Ядром A (Target) и Ядром B (Product)

В отличие от семейств F1-F18 (Ядра A и B), метасемейство F19 **не делится по ролям** Предприниматель/Инженер/Менеджер. Вместо этого оно организовано по функциональным областям.

## Связь с ядрами

```
┌─────────────────────────────────────────────────────────────────┐
│                         ЯДРО A (Target)                         │
│  A1.Среда → A2.Впечатлённый человек ← A3.ORA                   │
│                                          │                      │
├──────────────────── F19 (этот раздел) ───┼──────────────────────┤
│                                          ▼                      │
│                         ЯДРО B (Product)                        │
│  B1.Организатор ORA → B2.Приложение ← B3.Planora               │
└─────────────────────────────────────────────────────────────────┘
```

**Главный мост:** ORA (A3) использует Приложение (B2) как инструмент цифрового контура.

## Детали подразделов

### 0.1. Knowledge-Base-Logic

**Ключевой вопрос:** Как организованы знания?

Фундаментальные основы организации хранилища: модель 19 семейств документов, принципы, терминология. "Конституция" репозитория.

**Ключевые документы:**
- `document-families.md` — **главный документ**: модель 19 семейств (9+9+1)
- `glossary.md` — глоссарий (служебные термины + предметная область)
- `system-identification.md` — правила определения систем

### 0.2. Repository-Processes

**Ключевой вопрос:** Как работать с хранилищем?

Операционные процессы: структура папок, стандарты оформления, workflows создания документов.

**Ключевые документы:**
- `structure.md` — структура папок (двухъядерная модель)
- `standards.md` — стандарты оформления
- `document-creation.md` — процесс создания документов

### 0.3. Plans-and-Meetings

**Ключевой вопрос:** Что запланировано?

Планирование развития хранилища: roadmap, протоколы совещаний, координация.

### 0.4. AI-Reports

**Ключевой вопрос:** Какое состояние хранилища?

Автоматически генерируемые отчёты AI о качестве, покрытии, соответствии system-identification.md.

### 0.9. Inbox

**Ключевой вопрос:** Что нужно обработать?

Inbox для новых идей, черновиков и материалов до классификации по семействам.

### 0.99. Archive

**Ключевой вопрос:** Что было раньше?

Хранилище устаревших документов для сохранения истории.

## Принципы работы с разделом

1. **SSOT** — каждая сущность имеет одно место определения
2. **Метаданные обязательны** — YAML frontmatter для всех документов
3. **Разделение ролей** — люди: стратегия, AI: рутина
4. **Инкрементальное развитие** — не создавай документы "на будущее"
