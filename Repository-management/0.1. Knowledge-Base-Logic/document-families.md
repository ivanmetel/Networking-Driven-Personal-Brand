# Модель семейств документов

## Предварительные требования

> **ОБЯЗАТЕЛЬНО:** Для эффективной работы с этим хранилищем требуется понимание основ системного мышления.
>
> **Рекомендуемый материал:** ["Введение в системное мышление"](https://docs.system-school.ru/ru/personal/systems-thinking-introduction/preface) — руководство, охватывающее:

> - Роли и ролевое поведение

> - Системное моделирование

## Что такое семейство документов

**Семейство документов** — это категория документов, объединённых по двум критериям:

1. **Система** (System) — о какой части проекта документ
2. **Роль** (Role) — какой аспект системы описывает

Семейство — это не физическая папка, а **"взгляд" на знания** проекта. Один документ принадлежит одному семейству.

## Часть I: Базовая модель 3×3 (9 семейств)

SRD-метод организует знания через **таблицу 3×3**, которая определяет **9 базовых семейств**.

### Три системы (строки)

> **Принцип терминологии:** Основные термины систем берутся из FPF и системного мышления.
>
> **Подробное руководство:** [Принципы определения систем](system-identification.md)

| Система | Английское название (FPF) | Вопрос | Системное описание |
|---------|--------------------------|--------|-------------------|
| **Надсистема** | **Suprasystem** | В какую систему встроится? | Система, **частью которой** является целевая система |
| **Целевая система** | **System-of-Interest (SoI)** | Что создаём? | Система, которую проект создаёт. **Это результат, не процесс** |
| **Система создания** | **Constructor** | Кто/как создаёт? | Система, которая создаёт целевую систему |

### Три роли (столбцы)

| Роль | Английское название | Вопрос | Описание по FPF (A.7) |
|------|---------------------|--------|-----------------------|
| **Предприниматель** | Meaning | Зачем? | Интенсионал системы: цели, назначение, ценность |
| **Инженер** | Architecture | Как устроено? | Структурное описание: компоненты, связи, интерфейсы |
| **Менеджер** | Operations | Как работает? | Функциональное описание: процессы, workflow, поведение |

### Визуализация базовой таблицы 3×3

```
                     Предприниматель  Инженер           Менеджер
                     (Зачем?)         (Как устроено?)   (Как работает?)
┌────────────────────────────────────────────────────────────────────┐
│ Suprasystem        1                2                 3            │
│ (Надсистема)       Контекст         Окружение         Взаимодействие│
├────────────────────────────────────────────────────────────────────┤
│ System-of-Interest 4                5                 6            │
│ (Целевая система)  Требования       Архитектура       Реализация   │
├────────────────────────────────────────────────────────────────────┤
│ Constructor        7                8                 9            │
│ (Создающая сист.)  Принципы         Платформа         Команда      │
└────────────────────────────────────────────────────────────────────┘
```

### Описание 9 базовых семейств

| № | Система × Роль | Краткое название | Назначение |
|---|----------------|------------------|------------|
| **1** | Надсистема × Предприниматель | Контекст и ценность | Зачем надсистеме нужна целевая система |
| **2** | Надсистема × Инженер | Окружение и интерфейсы | Структура внешней среды |
| **3** | Надсистема × Менеджер | Взаимодействие и партнёрства | Поведение внешней среды |
| **4** | Целевая × Предприниматель | Требования и ценность | Что должен делать продукт |
| **5** | Целевая × Инженер | Архитектура продукта | Как устроен продукт внутри |
| **6** | Целевая × Менеджер | Реализация и процессы | Как продукт работает |
| **7** | Создающая × Предприниматель | Принципы и экономика | Бизнес-модель, ценности команды |
| **8** | Создающая × Инженер | Платформа и инструменты | Tech stack, инфраструктура |
| **9** | Создающая × Менеджер | Команда и методология | Процессы, роли, ритм работы |

## Часть II: Двухъядерная структура (18 семейств)

В данном хранилище используется **двухъядерная структура**: два независимых ядра, каждое из которых содержит свои 9 семейств по модели 3×3.

### Ядро A: Target (Целевая система — Впечатлённый человек)

Документация о создании **впечатлений у участников мероприятий**.

| Код | Система | Описание |
|-----|---------|----------|
| **A1** | Среда человека (Suprasystem) | Семья, сообщество, город, инфосреда |
| **A2** | Впечатлённый человек (SoI) | Человек с повышенной впечатлённостью |
| **A3** | ORA (Constructor) | Мероприятие-как-продукт |

**Семейства ядра A:**

| Семейство | Система × Роль | Содержание |
|-----------|----------------|------------|
| A1.1 | Среда × Предприниматель | Почему человеку нужны мероприятия |
| A1.2 | Среда × Инженер | Компоненты среды: семья, сообщество, город |
| A1.3 | Среда × Менеджер | Поведение человека в среде |
| A2.1 | Человек × Предприниматель | Определение впечатлённости, критерии успеха |
| A2.2 | Человек × Инженер | Компоненты: эмоции, связи, знания |
| A2.3 | Человек × Менеджер | Путь участника: до → во время → после |
| A3.1 | ORA × Предприниматель | Философия создания мероприятий |
| A3.2 | ORA × Инженер | Структура ORA, пакеты описаний |
| A3.3 | ORA × Менеджер | Процесс создания и проведения |

### Ядро B: Product (Наша система — Приложение)

Документация о создании **Приложения** для поддержки ORA.

| Код | Система | Описание |
|-----|---------|----------|
| **B1** | Организатор ORA (Suprasystem) | Надсистема для нашей системы: организатор + площадка + инструменты |
| **B2** | Приложение (SoI) | Наша система: платформа для ORA |
| **B3** | Компания Planora (Constructor) | Система создания нашей системы: ИТ-служба + ORA-студия |

**Семейства ядра B:**

| Семейство | Система × Роль | Содержание |
|-----------|----------------|------------|
| B1.1 | Организатор ORA × Предприниматель | Ценность для организатора |
| B1.2 | Организатор ORA × Инженер | Компоненты: организатор, площадка, логистика |
| B1.3 | Организатор ORA × Менеджер | Workflow организатора ORA |
| B2.1 | Приложение × Предприниматель | Требования, User Stories |
| B2.2 | Приложение × Инженер | Компоненты, API, данные |
| B2.3 | Приложение × Менеджер | Гайды, сценарии использования |
| B3.1 | Компания Planora × Предприниматель | Миссия, бизнес-модель |
| B3.2 | Компания Planora × Инженер | ИТ-служба + ORA-студия |

## Часть III: Семейство F0 — Метасистема и мосты

### Назначение F0

**F0 (Repository-management)** — особое семейство, которое:

1. **Управляет хранилищем** — онтология, структура, стандарты
2. **Связывает два ядра** — описывает мосты между ядром A и ядром B

F0 НЕ входит в таблицы 3×3 ядер, а является **метасистемой**, управляющей обоими ядрами.

### Структура F0

| Подраздел | Назначение |
|-----------|------------|
| 0.1. Knowledge-Base-Logic | Принципы организации, модель семейств, терминология, глоссарий |
| 0.2. Repository-Processes | Стандарты оформления, workflows, валидация |
| 0.3. Plans-and-Meetings | Планирование работы над хранилищем |
| 0.4. AI-Reports | Отчёты AI-агентов, автоанализ |
| 0.9. Inbox | Несортированные материалы, идеи для обработки |
| 0.99. Archive | Устаревшие и архивные документы |

### Мосты между ядрами

#### Ключевая связь ядер

```
A3.Constructor (ORA) ←──использует──→ B2.System-of-Interest (Приложение)
```

ORA использует Приложение как инструмент цифрового контура для:
- Регистрации участников
- Продажи билетов
- Коммуникаций
- Сбора данных
- Пост-ивент взаимодействия

#### Визуализация связи ядер

```
┌─────────────────────────────────────────────────────────────────┐
│                         ЯДРО A (Target)                         │
│                                                                 │
│  A1.Среда → A2.Впечатлённый человек ← A3.ORA                    │
│                                          │                      │
│                                          │ использует           │
├──────────────────────── F0.Bridge ───────┼──────────────────────┤
│                                          ▼                      │
│                         ЯДРО B (Product)                        │
│                                                                 │
│  B1.Создатель ORA → B2.Приложение ← B3.Planora                  │
└─────────────────────────────────────────────────────────────────┘
```

#### Роли на мосту между ядрами

| Роль | В ядре A | В ядре B |
|------|----------|----------|
| **ORA-студия** | Разрабатывает методики ORA (A3) | Внедряет Приложение (B3) |
| **Организатор** | Создаёт ORA (A3) | Использует Приложение (B1) |
| **Участник** | Получает впечатления (A2) | Взаимодействует через Приложение |

#### Цепочка создания ценности

```
Planora → Приложение → Создатель ORA → ORA → Впечатлённый человек → Среда
  (B3)      (B2)          (B1/A3)      (A3)        (A2)             (A1)
```

**Описание звеньев цепочки:**

| Звено | Вход | Выход | Ценность |
|-------|------|-------|----------|
| Planora → Приложение | Потребности организаторов | Работающее Приложение | Инструмент для создания ORA |
| Приложение → Создатель | Приложение как инструмент | Цифровой контур для ORA | Автоматизация, данные |
| Создатель → ORA | Приложение + площадка + контент | Проведённое мероприятие | Опыт для участников |
| ORA → Человек | Человек в исходном состоянии | Впечатлённый человек | Повышенная впечатлённость |
| Человек → Среда | Новые впечатления, связи | Изменения в поведении | Распространение эффекта |

#### Интеграция ORA и Приложения

Приложение — часть цифрового контура ORA на всех этапах:

**До мероприятия:**

| Функция | Что делает Приложение |
|---------|----------------------|
| Анонс | Страница события, описание, расписание |
| Регистрация | Форма, сбор данных участников |
| Билеты | Продажа, типы билетов, скидки |
| Коммуникации | Подтверждения, напоминания |

**Во время мероприятия:**

| Функция | Что делает Приложение |
|---------|----------------------|
| Check-in | Отметка прихода, бейджи |
| Программа | Расписание, навигация |
| Нетворкинг | Список участников, контакты |
| Обратная связь | Опросы в реальном времени |

**После мероприятия:**

| Функция | Что делает Приложение |
|---------|----------------------|
| Follow-up | Материалы, записи, презентации |
| Обратная связь | NPS, отзывы |
| Сообщество | Чаты, группы |
| Аналитика | Отчёты для организатора |

**Поток данных:**

```
Участник → [регистрация] → Приложение → [данные] → Организатор
                                ↓
                          [аналитика]
                                ↓
                    Улучшение следующей ORA
```

**Точки интеграции:**

1. **Приложение → ORA:** данные участников, статус регистрации
2. **ORA → Приложение:** программа, контент, обновления
3. **Участник → Приложение:** регистрация, платёж, обратная связь
4. **Приложение → Участник:** уведомления, материалы

#### Метрики по цепочке ценности

| Звено | Метрика |
|-------|---------|
| Planora → Приложение | Функциональность, стабильность |
| Приложение → Создатель | Adoption, retention организаторов |
| Создатель → ORA | Количество и качество мероприятий |
| ORA → Человек | Впечатлённость (W, W+Δ) |
| Человек → Среда | Рекомендации, повторное участие |

## Алгоритм определения семейства

```
1. ЭТО МЕТАСИСТЕМА?
   ├─ Управление хранилищем, связи ядер → F0 (Repository-management)
   └─ Нет → продолжить

2. О КАКОМ ЯДРЕ документ?
   ├─ О впечатлениях, участниках, мероприятиях → Ядро A (Target)
   └─ О приложении, организаторах, Planora → Ядро B (Product)

3. О КАКОЙ СИСТЕМЕ в ядре?
   ├─ О внешней среде → Suprasystem (X1)
   ├─ О целевой системе → System-of-Interest (X2)
   └─ О создателе → Constructor (X3)

4. КАКОЙ АСПЕКТ описывает?
   ├─ Зачем? Цели? Ценность? → Предприниматель (.1)
   ├─ Как устроено? Структура? → Инженер (.2)
   └─ Как работает? Процессы? → Менеджер (.3)

5. СЕМЕЙСТВО = Ядро + Система + Роль
   Примеры: A2.3, B1.1, B3.2
```

## Критерии качества описания (FPF E.10.D2)

Каждый документ должен соответствовать **I/D/S-дисциплине**:

| Критерий | Вопрос | Признак качества |
|----------|--------|------------------|
| **Интенсионал (I)** | Что это *по сути*? | Определение через род и видовое отличие |
| **Дескрипция (D)** | Что *наблюдаемо*? | Конкретные, проверяемые характеристики |
| **Спецификация (S)** | Что *требуется*? | Тестируемые, верифицируемые критерии |
