# Анализ репозитория Planora

## 1. Общие сведения
**Репозиторий:** `LariKing77/Planora`
**Цель анализа:** Изучить архитектурные решения и структуру папок для применения лучших практик в NDPB.

## 2. Структура файловой системы
В отличие от стандартного шаблона SRD, Planora использует кастомизированную структуру верхнего уровня:

*   **`Systems-Description/`**: Основная папка с контентом (вместо привычной `content/`).
*   **Двухъядерная организация:** Внутри `Systems-Description` находятся две директории верхнего уровня:
    1.  `A.Target`
    2.  `B.Product`

## 3. Описание систем (Dual Core)

### Ядро A: Target (Целевая система)
Фокус на изменении состояния человека (Конечный бенефициар).
*   **Suprasystem:** `A1.Suprasystem-Human-Environment` (Среда обитания человека).
*   **System-of-Interest:** `A2.System-of-Interest-Impressed-Human` (Впечатлённый человек).
*   **Constructor:** `A3.Constructor-ORA` (Ивент-продукт ORA).

### Ядро B: Product (Инструментальная система)
Фокус на техническом средстве создания.
*   **Suprasystem:** `B1.Suprasystem-ORA-Organization` (Организация ивента).
*   **System-of-Interest:** `B2.Target-system-App` (Приложение Planora).
*   **Constructor:** `B3.Constructor-Planora` (Команда/Компания Planora).

## 4. Наблюдения
1.  **Разделение ответственности:** Четко разведены "Софт" (Ядро B) и "Польза" (Ядро A). Приложение является лишь *инструментом* для Организатора, который создает *Ивент*, который меняет *Человека*.
2.  **Именование папок:** Используются префиксы (A1, A2, B1...) для жесткого упорядочивания.
3.  **Масштабируемость:** Такая структура позволяет описывать сложные экосистемные связи, где один продукт является конструктором для другой системы.
