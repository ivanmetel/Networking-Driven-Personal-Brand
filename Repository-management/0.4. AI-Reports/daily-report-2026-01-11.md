## Ключевые различия

### 1. Число систем

**Planora (2 системы):**
- **Система A (Target):** Впечатлённый человек — то, что создаёт бизнес
- **Система B (Product):** Приложение Planora — инструмент создания

**NDPB (1 система):**
- **System-of-Interest:** Одна целевая система (требуется уточнение)

> **Анализ FPF (A.1):** Planora описывает холонический паттерн, где продукт (система B) используется для создания результата (система A). NDPB описывает только продукт без явного разделения на продукт и результат.

### 2. Мосты между ядрами

**Planora:**
- ORA (A3) использует Приложение (B2) как инструмент
- Явно описаны связи между Ядром A и Ядром B

**NDPB:**
- Мосты между ядрами не применимы (одно ядро)
- Связи описываются через `related` в frontmatter

### 3. Назначение

**Planora:**
- Сервис, организующий встречи (ORA)
- Продукт (приложение) — инструмент для создания сервиса
- Фокус на впечатлённом человеке (результат)

**NDPB:**
- Требуется уточнение характера проекта
- Возможные варианты:
  - Прямой продукт (приложение, сервис)
  - Платформа
  - Сервис с двухъядерной структурой (как Planora)

---

## Сценарии выбора структуры

### Когда использовать одно ядро (NDPB-style)

✅ **Подходит, если:**
- Создаёте один продукт/приложение
- Продукт — конечная ценность для пользователя
- Нет явного разделения на инструмент и результат

**Примеры:**
- Финтех-приложение (FinTracker)
- E-commerce платформа
- SaaS-продукт
- Мобильное приложение

### Когда использовать два ядра (Planora-style)

✅ **Подходит, если:**
- Создаёте сервис, который использует продукт
- Есть чёткое разделение: инструмент (продукт) vs результат (впечатление)
- Описываете экосистему из взаимосвязанных систем

**Примеры:**
- Сервис организации мероприятий (Planora)
- Платформа для создания курсов
- Маркетплейс с фокусом на продавцов/покупателей
- EdTech-платформа (система обучения + приложение)

---

## Вопросы к NDPB

Для выбора оптимальной структуры NDPB необходимо ответить:

### 1. Что является основной ценностью?

- [ ] Продукт/приложение → **Одно ядро**
- [ ] Результат использования продукта → **Два ядра**

### 2. Есть ли разделение на инструмент и результат?

- [ ] Нет → **Одно ядро**
- [ ] Да → **Два ядра**

### 3. Кто целевой пользователь?

- [ ] Пользователь продукта → **Одно ядро**
- [ ] Участник процесса/явления → **Два ядра**

### 4. Какое название для NDPB?

- **Текущее название:** Networking-Driven Personal Brand
- **Анализ названия:**
  - "Personal Brand" — результат (впечатлённый человек)
  - "Networking-Driven" — процесс создания
- **Гипотеза:** Возможно, NDPB имеет двухъядерную структуру:
  - **Ядро A (Target):** Personal Brand (впечатлённый человек)
  - **Ядро B (Product):** Инструменты для создания (приложение, методология)

## Рекомендации

### Вариант 1: Оставить одно ядро (текущая структура)

**Если NDPB — прямой продукт:**
- Продолжить использовать F0 + F1-F9
- Уточнить System-of-Interest в PROJECT_DESCRIPTION.md
- Создать документы по 9 семействам

**Плюсы:**
- Проще структура
- Быстрее разворачивание
- Меньше документов для поддержания

**Минусы:**
- Ограниченная выразительность для сложных экосистем
- Может не отражать полный цикл создания ценности

### Вариант 2: Перейти на два ядра (Planora-style)

**Если NDPB — сервис с инструментами:**
- Создать структуру с 19 семействами
- Разделить на Ядро A (Target) и Ядро B (Product)
- Использовать F19 вместо F0

**Плюсы:**
- Глубокое описание экосистемы
- Явные мосты между системами
- Соответствует сложным бизнес-моделям

**Минусы:**
- Более сложная структура
- Больше документов для поддержки
- Требует чёткого определения двух систем
