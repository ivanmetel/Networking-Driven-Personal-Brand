# Процессы работы с хранилищем

## Назначение документа

Описание основных рабочих процессов (workflows) для работы с хранилищем знаний.

## Основные процессы

### 1. Создание нового документа

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  Определить │───▶│   Создать   │───▶│  Заполнить  │───▶│  Создать PR │
│   место     │    │    файл     │    │  контент    │    │   + Review  │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

**Шаги:**

1. **Определить место в SRT-таблице**
   - Выбрать систему (0-3)
   - Выбрать роль (Предприниматель/Инженер/Менеджер)
   - Проверить, нет ли похожего документа

2. **Создать файл**
   - Путь: `content/N.System/N.M.Role/filename.md`
   - Использовать шаблон из README папки
   - Заполнить frontmatter

3. **Заполнить контент**
   - Следовать стандартам оформления
   - Указать связи с существующими документами
   - Применить релевантные FPF-принципы

4. **Создать PR**
   - Описательный commit message
   - Запросить review
   - Дождаться approve

**См. также:** [Создание документов](document-creation.md)

### 2. Редактирование документа

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   Создать   │───▶│  Внести     │───▶│  Обновить   │───▶│  Создать PR │
│   ветку     │    │  изменения  │    │  метаданные │    │   + Review  │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

**Шаги:**

1. Создать feature-ветку
2. Внести изменения
3. Обновить `updated` в frontmatter
4. Проверить связи (related)
5. Создать PR с описанием изменений

### 3. Рефакторинг структуры

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  Анализ     │───▶│  Планирова- │───▶│ Выполнение  │───▶│  Валидация  │
│  структуры  │    │    ние      │    │ изменений   │    │   связей    │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

**Признаки необходимости рефакторинга:**
- Документ охватывает несколько ячеек матрицы
- Дублирование информации
- Нарушены связи между документами
- Противоречия с FPF-принципами

**Шаги:**

1. Проанализировать текущую структуру
2. Составить план изменений
3. Обновить документы по одному
4. Проверить все входящие/исходящие ссылки
5. Обновить related во всех затронутых документах

### 4. Review процесс

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   PR        │───▶│  Автопро-   │───▶│  Ручной     │───▶│   Merge     │
│  создан     │    │   верки     │    │   review    │    │             │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

**Чек-лист для reviewer:**

- [ ] Frontmatter заполнен корректно
- [ ] Документ в правильной папке
- [ ] Связи (related) актуальны
- [ ] Нет дублирования с существующими документами
- [ ] Соответствует стандартам оформления
- [ ] FPF-принципы применены корректно

### 5. Архивация документа

**Когда архивировать:**
- Документ устарел
- Информация перенесена в другой документ
- Функционал удалён

**Шаги:**

1. Изменить `status: archived` в frontmatter
2. Добавить причину архивации в начало документа
3. Обновить ссылки в связанных документах
4. НЕ удалять файл (для истории)

## Работа с ветками

### Naming convention

```
feature/add-api-documentation
fix/broken-links-in-management
refactor/reorganize-target-system
docs/update-glossary
```

### Основные ветки

| Ветка | Назначение |
|-------|------------|
| `main` | Стабильная версия |
| `feature/*` | Новые документы/функции |
| `fix/*` | Исправления |
| `refactor/*` | Рефакторинг |

## Автоматизация

### Pre-commit проверки

- Валидация YAML frontmatter
- Проверка существования файлов в `related`
- Проверка соответствия пути и system/role

### CI/CD

- Линтинг Markdown
- Проверка битых ссылок
- Генерация индексов (опционально)
