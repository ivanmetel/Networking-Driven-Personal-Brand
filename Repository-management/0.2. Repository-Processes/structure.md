# Структура хранилища

## Назначение документа

Описание архитектуры хранилища знаний, организации папок и файлов. Хранилище использует **двухъядерную модель** с 19 семействами документов.

## Двухъядерная модель

Хранилище организовано вокруг **двух системных ядер**, связанных мостом:

```
┌─────────────────────────────────────────────────────────────────┐
│                         ЯДРО A (Target)                         │
│                                                                 │
│  A1.Среда → A2.Впечатлённый человек ← A3.ORA                   │
│    (надсистема)    (целевая система)    (создатель)            │
│                                          │                      │
│                                          │ использует           │
├──────────────────────── F19.Bridge ──────┼──────────────────────┤
│                                          ▼                      │
│                         ЯДРО B (Product)                        │
│                                                                 │
│  B1.Организатор ORA → B2.Приложение ← B3.Компания Planora      │
│  (надсистема для      (наша система)   (система создания       │
│   нашей системы)                        нашей системы)         │
└─────────────────────────────────────────────────────────────────┘

        ┌───────────────────────────────────────────────┐
        │           F19. Repository-management          │
        │         (Метасистема + мосты между ядрами)    │
        └───────────────────────────────────────────────┘
```

## Цепочка создания ценности

```
Planora → Приложение → Организатор ORA → ORA → Впечатлённый человек → Среда
```

> **Терминология:** См. [glossary.md](../0.1.%20Knowledge-Base-Logic/glossary.md) для определений всех систем.

## Структура ядра

Каждое ядро имеет три системы, каждая система — три роли:

```
{Ядро}.{Система}/
├── README.md                     # Обзор системы
├── {Код}.1.Предприниматель/      # Зачем? (Meaning)
│   ├── README.md
│   └── [документы].md
├── {Код}.2.Инженер/              # Как устроено? (Architecture)
│   ├── README.md
│   └── [документы].md
└── {Код}.3.Менеджер/             # Как работает? (Operations)
    ├── README.md
    └── [документы].md
```

## Принципы именования

### Папки

| Элемент | Формат | Пример |
|---------|--------|--------|
| Ядро | `{Буква}.{Название}` | `A.Target`, `B.Product` |
| Система | `{Код}.{Тип}` | `A2.System-of-Interest` |
| Роль | `{Код}.{N}.{Роль}` | `A2.1.Предприниматель` |

### Файлы

| Правило | Пример |
|---------|--------|
| Нижний регистр | `api-design.md` |
| Дефис как разделитель | `user-authentication.md` |
| Описательное имя | `database-schema.md` |
| Расширение `.md` | `requirements.md` |

### Примеры путей

```
Repository-management/0.1. Knowledge-Base-Logic/glossary.md
Company-Content/A.Target/A2.System-of-Interest/A2.2.Инженер/persona-model.md
Company-Content/B.Product/B2.System-of-Interest/B2.2.Инженер/api-schema.md
Company-Content/B.Product/B3.Constructor/B3.3.Менеджер/ci-cd-pipeline.md
```

## FPF-связь

Структура отражает паттерны FPF:

| Паттерн | Применение в структуре |
|---------|------------------------|
| **A.1 Holonic Foundation** | Каждая папка — холон (целое и часть) |
| **A.1.1 Bounded Context** | Каждая папка — ограниченный контекст |
| **A.2 Role Taxonomy** | Роли Предприниматель/Инженер/Менеджер |
| **A.7 Strict Distinction** | Чёткое разделение типов контента |
